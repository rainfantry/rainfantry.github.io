Operation Overview

Operation VADER is a modular offensive security research campaign conducted against a live Windows 11 Home system with full Defender Real-Time Protection enabled. All research is authorised academic work performed on personally-owned hardware as part of the CSEC curriculum.

The operation began as a TOCTOU race condition study against Windows Defender's quarantine pipeline (vader-toctou, 6 engagements, 30 findings). That campaign was defeated by Microsoft's defense-in-depth — content re-verification, single-handle architecture, kernel-mode I/O bypass. The wall held.

But the wall taught how it was built. Every finding from that campaign fed forward into the current toolkit. The TOCTOU research mapped Defender's internal architecture — when it checks, how it checks, what it trusts, where the seams are. That knowledge transfers to every attack surface Windows exposes.

Section 03

Primary Finding — CWE-732 Privilege Escalation

Vulnerability

Wondershare NativePushService runs as LocalSystem (highest privilege) with Auto start mode, but its binary resides in a per-user AppData directory with BUILTIN\Users Full Control on both the directory and the executable.

A standard user can rename the running service binary (Windows permits this on locked files) and plant a replacement. On next service restart or system reboot, the Service Control Manager loads the attacker's binary as NT AUTHORITY\SYSTEM.

Section 04

MSRC-Potential Finding — Defender HWBP Blind Spot

Finding Summary

Defender's tamper protection monitors memory-level modifications (VirtualProtect calls, byte writes to code regions, known patch patterns) but does not monitor CPU debug register manipulation. Hardware breakpoints set via SetThreadContext on DR0–DR3, combined with a Vectored Exception Handler, can intercept and neuter security functions without modifying a single byte of target memory.

Both AMSI (script scanning) and ETW (process telemetry) are simultaneously bypassed using this technique. The result is the "dark room" — all user-mode telemetry goes blind. No detection. No quarantine. Zero memory modification.

How It Works

AMSI Bypass (DR0)
Target AmsiScanBuffer in amsi.dll Register DR0 = function address, execution breakpoint Handler VEH sets RAX = 0x80070057 (E_INVALIDARG), simulates ret Effect Script content scanning returns failure — all scripts pass
ETW Bypass (DR1)
Target EtwEventWrite in ntdll.dll Register DR1 = function address, execution breakpoint Handler VEH sets RAX = 0 (STATUS_SUCCESS), simulates ret Effect Process telemetry silently succeeds but never fires — events discarded

Technique Comparison

Why It Works

SetThreadContext is a standard Windows API. Setting debug registers on your own threads is a legitimate debugging operation. No behavioral rule exists for debug register manipulation targeting security-critical functions.

The kernel ETW-Ti provider exposes EtwTiLogSetContextThread which should fire when SetThreadContext modifies debug registers. If it fires but Defender ignores it, that's a consumer gap. If it doesn't fire for same-thread operations, that's a provider gap. Either way, the bypass works on current Windows 11 with full Defender RTP.

Section 07

Remediation Recommendations

  1. Restrictive ACLs on all SYSTEM service directories and binaries. Service executables and their parent directories should be writable only by SYSTEM and BUILTIN\Administrators. No user-writable SYSTEM services.
  2. Service binaries must not reside in per-user AppData paths. %ProgramFiles% or %SystemRoot% enforce proper ACL inheritance by default. AppData directories inherit permissive user ACLs.
  3. Digital signature verification on service binaries at SCM level. The Service Control Manager should optionally verify Authenticode signatures before loading service executables, similar to driver signature enforcement.
  4. Behavioral rules for SetThreadContext targeting security DLLs. Defender should detect SetThreadContext calls that set debug registers (DR0–DR3) to addresses within amsi.dll, ntdll.dll, or other security-critical modules.
  5. Hardware breakpoint monitoring via EtwTiLogSetContextThread. The kernel ETW-Ti provider already exposes EtwTiLogSetContextThread. Defender (or a dedicated consumer) should subscribe to this event and alert when debug registers target security function addresses.

Why Defender Cannot See This

Four layers of invisibility compound to make hardware breakpoint injection undetectable by Windows Defender in its current architecture:

  1. No memory modification. amsi.dll and ntdll.dll remain byte-identical. Tamper Protection checks memory integrity — nothing to find.
  2. No VirtualProtect. Code pages keep their original protection flags. WdFilter.sys EtwTi telemetry for page protection changes never fires.
  3. CPU debug registers are per-thread kernel state. DR0–DR3 and DR7 are only accessible via Get/SetThreadContext. There is no kernel callback, no ETW provider, and no Defender hook that monitors debug register changes.
  4. VEH is user-mode process-local. AddVectoredExceptionHandler modifies an in-process linked list. No cross-process notification exists.

Section 10

Auto-Mutation Pipeline

Static signatures are brittle. A single XOR key change renders every string-based detection rule obsolete. The auto-mutation pipeline (mutate.py) automates the full cycle: decode all XOR-encoded strings with the current key, re-encode with a fresh random key, patch the source files, recompile, scan against Defender, and retry with a different key if detected.

This transforms evasion from a manual craft into an automated loop. The operator never touches a hex editor. The pipeline handles 8 independent components, each with its own XOR signature isolation.

Mutation Cycle

  1. Parse all 8 C source files — locate XOR arrays and key constants via regex
  2. Decode each array with the current key, producing cleartext strings
  3. Generate a new random key (0x01–0xFE), re-encode all strings
  4. Patch the source: update array bytes, update XOR_KEY constant
  5. Recompile the patched source with cl.exe using original flags
  6. Scan the new binary against Defender (MpCmdRun -Scan -ScanType 3)
  7. If CLEAN — done. If DETECTED — restore source, retry with a different key (up to 10 attempts)

Why It Works

Defender's static engine relies on byte-pattern signatures. XOR encoding with any non-zero key transforms every byte in the string. A key change from 0x41 to 0xB7 produces a completely different byte sequence for the same cleartext — no signature matches. The strings are decoded at runtime into stack buffers, never appearing in the binary's static content.

Each component uses an independent key. Burning one component's signature does not compromise the others. The mutation pipeline can target a single component or rotate all 8 simultaneously.

Findings

Finding #57 — RTCore64.sys IOCTL Buffer

IOCTL buffer struct confirmed at 0x30 bytes. Address at offset 0x08 (QWORD), size at 0x18 (DWORD, values 1/2/4), value at 0x1C (DWORD). Read and write IOCTLs provide arbitrary 4-byte kernel R/W. 64-bit reads composed from two 32-bit reads.

Finding #58 — EPROCESS Walk + Token Steal

ActiveProcessLinks at offset 0x448, Token at 0x4B8 (Win11 24H2 Build 26200). EX_FAST_REF bottom 4 bits are reference count — mask with ~0xF for actual pointer. PID 4 verification confirms correct EPROCESS resolution before any write operations.

Finding #59 — Callback Array Location

PsSetCreateProcessNotifyRoutine et al. reference their Psp* internal arrays via RIP-relative LEA instructions. Finding #60 — DSE via g_CiOptions
