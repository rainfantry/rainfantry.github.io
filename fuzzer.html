mpengine.dll Fuzzing — Coverage-Guided Vulnerability Discovery

target methodology campaigns architecture path forward logs

// MISSION BRIEF

ASF Doctrine: Crash → Leak → Execute. To find crashes, you need fuzzing.

Target: mpengine.dll — the Microsoft Malware Protection Engine. It runs as NT AUTHORITY\SYSTEM. It parses every file format that touches the machine. It is network-reachable — send an email, drop a file on a share, visit a URL. The engine scans it before anything else does.

Every file format parser is attack surface. Every parser bug at SYSTEM privilege is a potential remote code execution. This is the campaign to find those bugs.

// TARGET ANALYSIS

What is mpengine.dll

mpengine.dll is the core scanning engine behind Microsoft Defender (formerly Windows Defender), Microsoft Security Essentials, and the Malicious Software Removal Tool. It ships on every Windows installation and runs as a privileged SYSTEM service via MsMpEng.exe.

Why this target

  • Privilege level: SYSTEM — a crash or code execution here is game over for the host
  • Attack surface: parses PE, scripts, documents, archives, ELF, Mach-O, custom containers, and dozens of other formats
  • Reachability: real-time protection triggers scans on file create, email receipt, web download, network share access — no user interaction required
  • Ubiquity: default on every Windows 10/11 installation. Billions of endpoints.

Scan pipeline

The Defender scan pipeline processes files through multiple stages before reaching a verdict. For the full breakdown of how the engine loads, initialises, and processes scan requests, see Rootkit Manual Ch.1 — Scan Pipeline.

FILE INPUT → FORMAT ID → PARSER SELECT → UNPACK / DECODE → SIGNATURE MATCH → HEURISTIC → VERDICT

Every parser that handles a format is a distinct attack surface. Container formats that trigger recursive unpacking (ZIP inside RAR inside ISO) multiply the code paths exercised per input.

Prior art

Tavis Ormandy (Google Project Zero) disclosed multiple critical vulnerabilities in mpengine.dll during 2017–2018, including CVE-2017-0290 (type confusion in the NScript JavaScript engine, RCE as SYSTEM via email/web). His work demonstrated that the engine's file format parsers are rich targets. Microsoft has hardened the engine since, but the attack surface — parsing arbitrary file formats at SYSTEM — remains structurally unchanged.

// METHODOLOGY

Fuzzing approach

Mutation-based black-box fuzzing. Take known file samples (seed corpus), apply random mutations, feed the mutated files to the scan engine, classify the result (clean / detection / crash / hang), and triage anything interesting.

SEED CORPUS → MUTATE → SCAN → TRIAGE → FEEDBACK

Seed corpus categories

CATEGORY FORMATS RATIONALE
PE EXE, DLL Primary scan target — most complex parser path
Script JS, VBS, PS1 NScript engine + AMSI integration — prior CVE target
Document DOC, PDF OLE/compound document parsers — historically buggy
Archive ZIP, RAR, CAB, ISO Recursive unpacking — multiplied code paths
Cross-platform ELF, Mach-O Less-fuzzed parsers — Defender scans these too

Mutation strategies

  • Bit flipping: random single-bit and multi-bit flips across the file
  • Byte insertion: random byte insertion/deletion to shift structure boundaries
  • Format-aware header mutation: targeted corruption of format headers (PE optional header, ZIP local file header, ELF program headers)
  • Magic byte corruption: corrupt format magic bytes (MZ, PK, 7fELF) to probe format detection fallback paths
  • Weighted selection: mutation type selected per-iteration based on file format — archive seeds get more header mutations, script seeds get more content insertion

For the theory behind mutation strategies and corpus design, see Field Manual Ch.5 — Fuzzing Theory and Ch.10 — Corpus & Dictionary Design.

// CAMPAIGN RESULTS

CAMPAIGN 1: PARALLEL FUZZING (100K)

ENGINE mpengine.dll v1.1.26050.11 ITERATIONS 100,000 WORKERS 4 parallel RATE 3.8 exec/sec DETECTIONS 19 (mostly script seeds — expected, not bugs) CRASHES 0 HANGS 0 DURATION ~7.3 hours

CAMPAIGN 2: DIFFERENTIAL FUZZING (10K)

ENGINE mpengine.dll v1.1.26050.11 ITERATIONS 10,000 WORKERS 1 (differential mode) RATE 2.1 → 1.3 exec/sec RATE NOTE degradation over time — engine resource accumulation CLEAN 10,000 DETECTIONS 0 CRASHES 0 HANGS 0 DURATION ~2.1 hours

Analysis

Zero crashes across 110,000 iterations confirms that blind mutation fuzzing at 3.8 exec/sec is insufficient to find bugs in a hardened target like mpengine.dll. The engine has been fuzzed extensively by Microsoft's internal teams and by external researchers (OSS-Fuzz, Project Zero). Finding new vulnerabilities requires either dramatically higher throughput (10,000+ exec/sec), coverage-guided feedback to evolve inputs toward unexplored code paths, or both.

The 19 detections in Campaign 1 are false positives from the fuzzer's perspective — script seeds with known-malicious patterns that survive mutation. These confirm the engine is scanning correctly, not that mutations found anything new.

The rate degradation in Campaign 2 (2.1 to 1.3 exec/sec) suggests the engine accumulates resources over long runs. This is a signal for snapshot-based fuzzing — start each iteration from a clean memory state.

Current limitation

The fuzzer invokes MpCmdRun.exe as a subprocess for each scan. This means every iteration pays the cost of process creation, engine initialisation, definition loading, and file I/O. At 3.8 exec/sec with 4 workers, each individual scan takes ~1 second — and most of that time is overhead, not actual parsing.

This is the primary bottleneck. See Path Forward for the fix.

// PATH FORWARD

What's needed to make this CVE-grade. Each step is a multiplier on the previous.

01

Direct mpengine.dll loading

Bypass MpCmdRun.exe entirely. Load mpengine.dll as a library, call the scan API directly. Eliminates process creation, engine init, and definition reload per iteration.

IMPACT: 50–500x speed improvement. Target: 200–2,000 exec/sec.

See Rootkit Manual Ch.5 — Automation & Engine Interfaces for the scan API surface.

02

Coverage-guided feedback

Instrument mpengine.dll with WinAFL or build a LibFuzzer harness. Track which code paths each input exercises. Evolve the corpus toward inputs that hit new basic blocks — especially in less-exercised format parsers.

IMPACT: finds bugs that blind mutation never reaches. Required for novel CVEs.

See Field Manual Ch.5 — Fuzzing Theory for coverage-guided fundamentals, and Ch.9 — Harness Engineering for harness construction.

03

WTF snapshot fuzzing

Snapshot the engine's memory state on Hyper-V at the point where it's about to parse a file. Replay from snapshot for each iteration — no process start, no init, no definition load. Combine with coverage feedback.

IMPACT: thousands of exec/sec. This is how Microsoft's own teams fuzz internally.

Compute target: RunPod RTX 5090 for parallelised snapshot execution.

04

Format-specific corpus engineering

Hand-crafted seeds targeting less-fuzzed parsers: container formats (ISO, VHD, WIM), nested archives (ZIP-in-RAR-in-CAB), recently-added format support, and edge-case file structures designed to exercise unusual code paths.

IMPACT: reaches parser code that random mutation from generic seeds never touches.

See Field Manual Ch.10 — Corpus & Dictionary Design for systematic corpus construction.

05

Crash triage pipeline

Automated exploitability assessment for any crashes found. WinDBG !exploitable classification, custom triage scripts for root cause analysis, automated minimisation to find the smallest input that reproduces the crash.

IMPACT: turns a crash into an actionable vulnerability report. Required for responsible disclosure via MSRC.

See Field Manual Ch.6 — Crash Triage for triage methodology and Ch.12 — Exploit Development for the crash-to-exploit pipeline.
