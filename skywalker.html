Section 01

Operation Overview

SkyWalker is a complete, independent fork of the VADER rootkit kill chain. Same architecture. Same phases. Completely different binary signatures. Every XOR key, every binary name, every compiled byte pattern is isolated from VADER. Zero shared static signatures between the two.

The purpose is operational redundancy. Defender signature databases eventually catch known static patterns. When VADER's XOR arrays are flagged and mutate.py exhausts 10 key-rotation attempts without a clean scan, SkyWalker deploys from cold standby with no shared forensic footprint. It's a second set of teeth behind a different face.

SkyWalker also serves as a parallel operations platform — two simultaneous targets, two independent toolsets, zero cross-contamination if one gets burned.

Section 02

Cold Standby Doctrine

A single toolset is a single point of failure. If Defender develops a behavioural signature for VADER's compiled output — even after XOR mutation — the entire kill chain is burned. SkyWalker exists so that "burned" never means "disarmed."

Signature Isolation

Every XOR key differs from VADER. Every binary name differs. Every compiled .text section has unique byte patterns. Defender cannot cross-reference one to detect the other.

Cold Activation

SkyWalker sits dormant. No network traffic. No running processes. No filesystem artifacts. Activated only when VADER fails 10 consecutive mutation cycles or parallel ops are required.

Parallel Operations

Two targets, two toolsets, zero cross-contamination. If Target A's forensics team recovers VADER artifacts, Target B's SkyWalker deployment is unaffected.

Diff-Test Baseline

When VADER introduces experimental features, SkyWalker serves as a known-good baseline for regression testing. Same architecture, different implementation — the delta is always isolated.

Section 05

Eclipse — AMSI + ETW Bypass

Eclipse is SkyWalker's combined AMSI + ETW bypass — equivalent to VADER's dark_room.exe but with different XOR keys and binary identity. The technique is identical: hardware breakpoints via SetThreadContext on DR0 (AmsiScanBuffer) and DR1 (EtwEventWrite), combined with a Vectored Exception Handler that intercepts the breakpoint and neuters the function.
