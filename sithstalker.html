Section 02

Theory — Why Indirect Syscalls

Every Windows API call eventually reaches an Nt* function in ntdll.dll. That function loads a System Service Number (SSN) into EAX, transfers to R10, and executes syscall. EDR products intercept this by hooking the first bytes of those stubs — replacing the clean mov r10, rcx; mov eax, SSN sequence with a JMP into their analysis engine.

Direct syscall skips ntdll entirely — you issue syscall from your own .text section. But the return address on the stack points outside ntdll. Modern EDR checks this via InstrumentationCallback. Your binary lights up like a flare.

Indirect syscall resolves the SSN at runtime, then jumps to a legitimate syscall; ret gadget inside ntdll. The return address looks like a normal ntdll call. No hooks triggered. No stack anomaly. The OS itself doesn't know the call didn't originate from ntdll.

Hell's Gate (am0nsec + smelly__vx, 2020)

Walk the PEB (__readgsqword(0x60)) → InMemoryOrderModuleList → second entry = ntdll → parse PE export table → hash each function name with DJB2 (seed 5381) → match target hash → extract SSN from bytes 4-5 of the stub pattern 4C 8B D1 B8 XX XX 00 00 → scan forward for 0F 05 C3 (syscall;ret gadget).

Halo's Gate (sektor7)

When EDR has overwritten the stub (first bytes are a JMP, not 4C 8B D1), look at neighboring exports ±20 positions. Find a clean stub, read its SSN, apply arithmetic offset (neighbor_ssn ± delta). Recovers up to 20 consecutively hooked functions.

Section 04

User-Mode Concealment

Three inline hooks on ntdll.dll intercept OS queries at the syscall boundary. The hooks are installed via cloak.dll, delivered system-wide through SetWindowsHookEx(WH_CBT, ..., 0). One command and every GUI process on the desktop is cloaked. Config strings are XOR-encrypted in the binary — zero plaintext in .rdata.

Process Hiding

Hooks NtQuerySystemInformation. Intercepts SystemProcessInformation (class 5) queries. Walks the SYSTEM_PROCESS_INFORMATION linked list and unlinks entries matching hidden process names by adjusting NextEntryOffset. Task Manager, tasklist, Process Explorer — all blind.

File Hiding

Hooks NtQueryDirectoryFile. Handles 5 information classes: FileBothDirectoryInformation, FileDirectoryInformation, FileFullDirectoryInformation, FileNamesInformation, FileIdBothDirectoryInformation. Unlinks matching entries from the directory buffer. dir, Explorer, everything built on FindFirstFile — blind.

Connection Hiding

Hooks NtDeviceIoControlFile. Filters MIB_TCPROW entries from TCP/UDP table IOCTLs by matching dwRemotePort against hidden port list. netstat -ano, TCPView, CurrPorts — C2 connection invisible.

System-Wide Delivery

cloak_loader.exe installs a global WH_CBT hook. Windows loads cloak.dll into every GUI process. DllMain(DLL_PROCESS_ATTACH) installs the 3 inline hooks per-process. The CBT proc is a passthrough — it exists only to trigger DLL injection.

Hook Engineering — The 16-Byte Fix

Initial hook patch size was 12 bytes. Nt* stubs on 24H2 include an 8-byte test byte ptr [SharedUserData+0x308], 1 instruction at offset 8. A 12-byte patch split this instruction — the remaining 4 bytes were interpreted as JMP displacement → ACCESS VIOLATION → Explorer crash. Fix: HOOK_PATCH_SIZE 12 → 16. Save and restore 16 bytes minimum. Confirmed stable across all GUI processes including Explorer.

Section 05

Kernel DKOM via BYOVD

User-mode concealment hides from applications. Kernel DKOM hides from the kernel itself. kernel_cloak.exe loads RTCore64.sys (CVE-2019-16098, signed by Micro-Star International) via SCM, obtains a device handle, and uses the driver's arbitrary physical memory read/write IOCTLs to walk and modify kernel structures directly.

Kernel Operations
  1. Load RTCore64.sys via sc create + sc start — signed driver, no DSE bypass needed
  2. Walk x64 4-level page tables (PML4 → PDPT → PD → PT) for VA-to-physical translation
  3. Locate EPROCESS for PID 4 (System) by scanning physical memory for known PID=4 pattern
  4. Auto-detect ActiveProcessLinks and UniqueProcessId offsets from System EPROCESS
  5. Walk ActiveProcessLinks doubly-linked list to find target process EPROCESS
  6. Unlink target: prev->Flink = target->Flink; next->Blink = target->Blink
  7. Target process vanishes from ALL kernel-level queries — Process Explorer, kernel debugger, everything
Driver: RTCore64.sys (40,688 bytes, Micro-Star International signed) CVE: CVE-2019-16098 — arbitrary read/write via IOCTLs Location: C:\Program Files (x86)\MSI Afterburner\RTCore64.sys Technique: EPROCESS ActiveProcessLinks unlink via physical memory R/W Result: Process invisible at kernel level — survives all user-mode enumeration

Mutation Pipeline

mutate.py rotates the XOR encryption key across all 13 hash arrays in gate_v2.h, rebuilds both engines, runs the 13/13 resolution test, and scans against Defender. If any step fails, it rolls back automatically. Every mutation produces a unique binary — no two compiled copies share static byte patterns.

Run test_gate.exe → verify 10/10 6. Run test_gate_v2.exe → verify 13/13 7. Scan both binaries with MpCmdRun.exe 8. Pass → commit new key. Fail → rollback to previous key. 9.
